# End-to-End ML CI/CD Pipeline Implementation Plan

**TL;DR:** Build a GitHub Actions CI/CD pipeline for the hybrid-log-analyzer that automates linting, testing, DVC-based reproducible training/evaluation, and CML-powered PR reporting. CI runs CPU-only (heavy GPU training stays in Colab); data and models are versioned via DVC with a GCS remote; Papermill drives parameterized notebook execution; CML posts metrics/plots as PR comments. The LLM component will use API keys (stored as secrets) for prompt engineering and DVC-tracked fine-tuned model artifacts from Colab.

## 2. DVC Initialization & Pipeline Definition

- Run `dvc init` and configure a GCS remote:
  ```bash
  dvc remote add -d gcs gs://<YOUR_BUCKET>/dvc-store
  ```
- Auth via service account: create a GCP service account with Storage Object Admin role, download JSON key, store contents as GitHub secret `GCP_SA_JSON`
- Define `dvc.yaml` with a multi-stage reproducible DAG:
  - **Stage `preprocess`**: `python src/preprocessing.py` тАФ reads raw logs from `data/raw/`, outputs `data/processed/`. Deps: `src/preprocessing.py`, `data/raw/`. Outs: `data/processed/`.
  - **Stage `train`**: `papermill notebooks/train_gat.ipynb outputs/train_output.ipynb -f configs/params.yaml -k python3 --log-output` тАФ trains GNN on CPU (small subset) or loads a pre-trained model. Deps: `notebooks/train_gat.ipynb`, `data/processed/`, `configs/params.yaml`. Outs: `models/gat_model.pt`.
  - **Stage `evaluate`**: `papermill notebooks/evaluate.ipynb outputs/evaluate_output.ipynb -k python3 --log-output` тАФ computes metrics, generates plots. Deps: `notebooks/evaluate.ipynb`, `models/gat_model.pt`, `data/processed/`. Metrics: `metrics.json` (cache: false). Plots: `plots/` (ROC curves, loss graphs).
- Define `configs/params.yaml` with all tunable hyperparameters (`learning_rate`, `GAT_attention_heads`, `batch_size`, `hidden_dim`, `epochs`, `dataset_path`, etc.)

---

## 3. Notebook Parameterization (Papermill)

- In each notebook (`train_gat.ipynb`, `evaluate.ipynb`, `finetune_llm.ipynb`), tag one cell with the `parameters` tag containing default values for all configurable variables
- Ensure no inter-dependent parameters within the tagged cell (Papermill injects values without re-evaluating expressions)
- Add guards in notebooks: detect environment (Colab vs. CI) via `os.environ.get('CI')` тАФ skip `drive.mount()` when running in CI, use local paths instead
- Install `ipykernel` and register the kernel in CI:
  ```bash
  python -m ipykernel install --user --name python3
  ```

## 4. Workflow: Lint (`lint.yaml`)

- **Trigger:** `on: [push, pull_request]`
- **Runner:** `ubuntu-latest`
- **Steps:**
  1. `actions/checkout@v5`
  2. `actions/setup-python@v5` with `python-version: '3.11'` and `cache: 'pip'`
  3. `pip install ruff mypy nbstripout`
  4. `ruff check src/ tests/` тАФ lint Python code
  5. `mypy src/` тАФ type checking
  6. `nbstripout --is-stripped notebooks/*.ipynb` тАФ verify notebook outputs stripped

---

## 5. Workflow: Test (`test.yaml`)

- **Trigger:** `on: [push, pull_request]`
- **Runner:** `ubuntu-latest`
- **Steps:**
  1. Checkout, setup Python (same as lint)
  2. `pip install -r requirements.txt`
  3. `pytest tests/ -v --tb=short` тАФ unit tests for preprocessing, model architecture, utility functions
  4. Optionally: a smoke test that runs a very small training batch to validate end-to-end flow

---

## 6. Workflow: Evaluate on PR (`evaluate.yaml`)

- **Trigger:** `on: pull_request: branches: [main]`
- **Runner:** `ubuntu-latest`
- **Steps:**
  1. `actions/checkout@v5` with `fetch-depth: 0` (needed for `dvc metrics diff`)
  2. `actions/setup-python@v5` (Python 3.11, pip cache)
  3. `treeverse/setup-dvc@v2`
  4. `iterative/setup-cml@v1`
  5. Authenticate to GCS: `google-github-actions/auth@v3` with `credentials_json: ${{ secrets.GCP_SA_JSON }}`
  6. `pip install -r requirements.txt && python -m ipykernel install --user --name python3`
  7. `dvc pull` тАФ fetch data and model artifacts from GCS
  8. `dvc repro evaluate` тАФ run only the evaluate stage (reuses cached training)
  9. Generate CML report:
     - `dvc metrics diff main --show-md >> report.md` тАФ metrics comparison table
     - `dvc plots diff --target plots/ main` тАФ generate diff plots
     - Append plots, ROC curves, loss graphs to `report.md`
     - `cml comment create report.md` with `REPO_TOKEN: ${{ secrets.GITHUB_TOKEN }}`
  10. `actions/upload-artifact@v4` тАФ upload executed notebook outputs and metrics

---

## 7. Workflow: Train on Main (`train.yaml`)

- **Trigger:** `on: push: branches: [main], paths: ['notebooks/**', 'src/**', 'configs/**', 'dvc.yaml']` + `workflow_dispatch` with input parameters (learning_rate, epochs, etc.)
- **Runner:** `ubuntu-latest` (CPU тАФ smoke training or eval only; real training in Colab)
- **Steps:**
  1. Checkout (fetch-depth: 0), setup Python, DVC, CML, GCS auth
  2. `pip install -r requirements.txt && python -m ipykernel install --user --name python3`
  3. `dvc pull`
  4. `dvc repro` тАФ execute the full pipeline (preprocess тЖТ train тЖТ evaluate)
  5. `dvc push` тАФ upload new model artifacts and cache to GCS
  6. Commit `dvc.lock` and `metrics.json` back to the repo (git add, commit, push using `github-actions[bot]`)
  7. Upload artifacts via `actions/upload-artifact@v4`
- **Note:** For CPU-only CI, the training notebook should either train on a tiny data subset (to validate the pipeline works) or skip training entirely and evaluate a pre-existing model. Full GPU training happens in Colab, and the user runs `dvc push` manually after.

---

## 8. Workflow: Scheduled Retrain (`retrain.yaml`)

- **Trigger:** `on: schedule: - cron: '0 2 * * 0'` (weekly, Sunday 2 AM UTC) + `workflow_dispatch`
- Same structure as `train.yaml`, but:
  - Pulls the latest data version
  - Passes `-p data_version latest` to Papermill
  - After successful retraining, commits updated `dvc.lock` and `metrics.json`
  - Could trigger a downstream notification (Slack webhook, email) on failure

---

## 9. Google Cloud & Secrets Configuration

- **GCP setup:**
  - Create a GCP project, enable Cloud Storage API and Google Drive API
  - Create a GCS bucket for DVC remote (e.g., `gs://hybrid-log-analyzer-dvc`)
  - Create a Service Account with `Storage Object Admin` role
  - Download JSON key тЖТ store as GitHub repository secret `GCP_SA_JSON`
- **API keys for LLM:**
  - Store OpenAI/Gemini API keys as secrets: `OPENAI_API_KEY`, `GEMINI_API_KEY`
  - Reference in workflows via `env:` blocks (never inline)
- **CML:** Uses `GITHUB_TOKEN` (auto-provided) for PR comments тАФ no PAT needed for CPU-only runners
- Add `gha-creds-*.json` to `.gitignore`

## 10. Colab тЖФ CI Synchronization Pattern

Since real GPU training happens in Colab interactively:

1. Developer trains in Colab, saves model to GCS (via `gcsfs` or `gsutil`) or adds to DVC
2. Developer runs `dvc push` from Colab to upload artifacts to GCS
3. Developer commits updated `dvc.lock` to Git
4. CI (evaluate workflow) picks up the new model via `dvc pull` and runs evaluation
5. CML posts comparison metrics on the PR

## Verification Checklist

1. **Lint workflow**: Push any Python change тЖТ verify ruff and mypy pass in Actions
2. **Test workflow**: Push change тЖТ verify pytest runs and passes
3. **Evaluate workflow**: Open a PR to main тЖТ verify DVC pulls data, evaluation runs, CML posts metrics comment on the PR
4. **Train workflow**: Merge to main тЖТ verify `dvc repro` runs, `dvc.lock` is updated and committed back
5. **Retrain workflow**: Manually trigger via workflow_dispatch тЖТ verify end-to-end execution
6. **DVC round-trip**: Run `dvc push` from Colab, then trigger CI `dvc pull` тЖТ verify artifacts match
7. **Secrets**: Verify no secrets leak in logs; check GCS auth works

## Key Decisions

| Decision | Rationale |
|---|---|
| **No Colab in CI** | Google Colab has no reliable headless API; CI runs on GitHub-hosted Ubuntu runners (CPU). Full GPU training stays interactive in Colab. |
| **GCS over Google Drive** | GCS has native service account auth, no quota limits, and first-class DVC support тАФ far more reliable in CI. |
| **Full DVC pipeline** | Using `dvc.yaml` with `dvc repro` for reproducible DAG execution, not just data versioning. |
| **Separate workflows** | 5 independent workflows (lint, test, evaluate, train, retrain) for fast feedback, cost control, and clear failure isolation. |
| **Papermill for notebooks** | Parameterized execution in CI with environment detection to skip Colab-specific code (like `drive.mount()`). |
| **CML for reporting** | Metrics diff and plots posted as PR comments тАФ no external dashboard needed. |
| **Service Account auth** | Using `google-github-actions/auth@v3` with JSON key secret (not OIDC) since OIDC tokens expire too quickly for potential long-running steps. |
