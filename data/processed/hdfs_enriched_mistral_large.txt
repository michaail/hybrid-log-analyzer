#### **5. Anomaly Detection Semantics**
- **Key Observations for Anomaly Detection:**
  1. **Isolated Allocation:**
     - If this log appears **without** subsequent logs for:
       - `BlockReport` (DataNode reporting the block).
       - `addStoredBlock` (DataNode confirming block storage).
       - `ReceivedBlockReport` (NameNode acknowledging the block).
     → **Potential anomaly:** The block was allocated but never written or stored (e.g., client failure, network partition).
  2. **Duplicate Allocations:**
     - Multiple `allocateBlock` logs for the **same block ID** may indicate:
       - **NameNode metadata corruption** (e.g., race condition in block assignment).
       - **Client retries** (e.g., due to timeouts).
  3. **Orphaned Blocks:**
     - If the block is allocated but the file is **never closed** (no `completeFile` log), it may become an orphaned block, wasting storage.
  4. **Replication Issues:**
     - If the block is allocated but **under-replicated** (e.g., due to DataNode failures), later logs may show `UnderReplicatedBlocks` warnings.

- **Labeling Rules (Benchmark Context):**
  - **Normal:** This log is part of a sequence where the block is:

    - Closed successfully (`completeFile`).
  - **Anomalous:** This log is part of a sequence where:
    - The block is allocated but **no further logs** appear (e.g., client crash).
